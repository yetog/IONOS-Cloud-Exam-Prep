### 3. Question Parser Script
`Scripts/parse_questions.py` — converts all 14 markdown sectional exam files into a TypeScript question bank (`src/data/questionBank.ts`) using a line-by-line parser.

**How it works:**
- Splits each file on `## Answer Key`
- Parses question blocks: `**N. Question?**` followed by `A. B. C. D.` option lines
- Parses answer key: `N. [A-D] — Explanation`
- Maps letter answers to 0-based index (`A=0, B=1, C=2, D=3`)
- Derives difficulty from question number (1–10=easy, 11–20=medium, 21–30=hard)
- Outputs valid TypeScript matching the `Question` interface

## The Chunking Problem — Full Diagnosis & Fix

### The Problem
After wiring 420 questions into the app, the production build produced a single 2,076 kB JavaScript bundle with this warning:

```
(!) Some chunks are larger than 500 kB after minification.
```

A 2 MB bundle means the browser must download, parse, and execute all of that JavaScript before the user sees anything. On a slow connection or mobile device, this causes a noticeable blank screen delay.

---

### Root Causes (in order of discovery)

#### Root Cause 1: No code splitting at all
The original `App.tsx` used static imports for all 14 pages:

```tsx
import Dashboard from "./pages/Dashboard";
import PracticeMode from "./pages/PracticeMode";
import SkillsPage from "./pages/SkillsPage";
// ... 11 more
```

Static imports are bundled together at build time regardless of whether the user ever visits that route. Every page — including the 511-line `PracticeMode` and 344-line `Dashboard` — was packed into one file.

#### Root Cause 2: Question bank loaded synchronously
The 420-question bank was imported statically in `sampleQuestions.ts`:

```ts
import { QUESTION_BANK } from '@/data/questionBank';
```

This meant 412 kB of question data was bundled into the initial load even though the user doesn't need questions until they click "Practice."

#### Root Cause 3: Entire lucide-react icon library imported twice
Both `SkillsPanel.tsx` and `SkillBar.tsx` had this import:

```ts
import { icons } from 'lucide-react';
```

The `icons` export is an object containing **every single icon** in the lucide library (~1,000+ icons, ~650 kB). They only needed ~20 icons for the skills UI, but imported all of them to do a dynamic lookup:

```ts
const IconComponent = icons[categoryInfo.icon as keyof typeof icons];
```

This alone accounted for the 669 kB `SkillsPage` chunk.

---

### The Fixes (in order applied)

#### Fix 1: Vendor chunk splitting via `vite.config.ts`
Use Vite's `manualChunks` function to isolate stable third-party libraries into their own cached chunks. The key insight: vendor libraries rarely change, so once a user's browser caches them, they don't re-download on each deploy.

```ts
// vite.config.ts
build: {
  rollupOptions: {
    output: {
      manualChunks(id) {
        if (id.includes('node_modules/react') || id.includes('node_modules/react-dom') ...) {
          return 'react-vendor';      // React core — cached long-term
        }
        if (id.includes('node_modules/framer-motion')) {
          return 'motion-vendor';     // Animation library — isolated
        }
        if (id.includes('node_modules/@radix-ui') ...) {
          return 'ui-vendor';         // Component primitives — isolated
        }
        if (id.includes('node_modules/howler')) {
          return 'howler-vendor';     // Audio — isolated
        }
        if (id.includes('/src/data/questionBank')) {
          return 'question-bank';     // Our data — lazy loaded separately
        }
      },
    },
  },
},
```

**Why use a function instead of an object?**
The object form of `manualChunks` maps exact package names, but Rollup resolves modules by file path (e.g., `node_modules/framer-motion/dist/framer-motion.js`). Using a function with `id.includes()` catches all files within a package reliably.

#### Fix 2: Route-based lazy loading via `React.lazy()`
Replace all static page imports with dynamic imports wrapped in `React.lazy()`. React only loads a page's JavaScript bundle when the user actually navigates to that route.

```tsx
// Before (loads everything upfront)
import PracticeMode from "./pages/PracticeMode";
import SkillsPage from "./pages/SkillsPage";

// After (loads only when route is visited)
const PracticeMode = lazy(() => import("./pages/PracticeMode"));
const SkillsPage = lazy(() => import("./pages/SkillsPage"));
```

Wrap all routes in `<Suspense>` to show a fallback while the chunk loads:

```tsx
<Suspense fallback={<div className="...">Loading...</div>}>
  <Routes>
    <Route path="/practice" element={<PracticeMode />} />
    ...
  </Routes>
</Suspense>
```

**Important:** Keep `Dashboard` as a static import since it's the first thing every user sees — no point lazy-loading it.

#### Fix 3: Async question bank loading via dynamic import
Remove the static import from `sampleQuestions.ts` and use `useEffect` + dynamic import in the `useQuestions` hook:

```ts
// useQuestions.ts
const [bankQuestions, setBankQuestions] = useState<Question[]>([]);

useEffect(() => {
  // Loads questionBank.js asynchronously after the page renders
  import('@/data/questionBank').then((m) => setBankQuestions(m.QUESTION_BANK));
}, []);

const allQuestions = useMemo(() => {
  return [...SAMPLE_QUESTIONS, ...bankQuestions, ...customQuestions];
}, [bankQuestions, customQuestions]);
```

**Result:** The dashboard renders immediately with 6 sample questions available. The 412 kB question bank loads in the background. By the time the user navigates to Practice Mode, all 420 questions are ready.

#### Fix 4: Replace dynamic icon lookups with static maps
Instead of importing the full `icons` bundle and doing a runtime string lookup, import only the icons actually used and create an explicit map:

```ts
// Before — imports ~650 kB of icons
import { icons } from 'lucide-react';
const IconComponent = icons[skill.icon as keyof typeof icons];

// After — imports only what's needed
import { Target, Calculator, BookOpen, BarChart3, ... } from 'lucide-react';
const ICON_MAP: Record<string, React.ElementType> = {
  Target, Calculator, BookOpen, BarChart3, ...
};
const IconComponent = ICON_MAP[skill.icon];
```

**Trade-off:** If a new icon name is added to `skills.ts`, it must also be added to `ICON_MAP`. This is a minor maintenance cost for a massive bundle size reduction.

### Key Lessons

1. **Static imports = eager bundling.** Everything you `import` at the top of a file is included in that file's chunk at build time, regardless of whether it's used on every page.

2. **`React.lazy()` + `<Suspense>` is the standard pattern for route-based splitting.** It requires zero changes to the page components themselves — only the router file changes.

3. **Dynamic imports (`import()`) work anywhere, not just with React.lazy.** You can use them in hooks, utility functions, or anywhere you want to defer loading.

4. **`manualChunks` as a function > as an object.** Rollup operates on file paths, not package names. The function form with `id.includes('node_modules/package-name')` is more reliable.

5. **The `icons` export from lucide-react is a bundle killer.** Always import named icons individually (`import { Target } from 'lucide-react'`). If you need dynamic icon rendering, build a local map of only the icons you actually use.

6. **Vendor chunks enable long-term browser caching.** React, framer-motion, and Radix UI don't change between your deploys. Isolating them means returning users load those chunks from cache instantly.
